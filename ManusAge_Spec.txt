# ManusAge — System Specification

## 1. Purpose
ManusAge is a GenAI system designed to estimate the age of handwritten ink and provide document intelligence using Retrieval-Augmented Generation (RAG), multi-agent orchestration, and automated evaluation. The system emphasizes transparency, accuracy, and engineering rigor by evaluating every answer using LLM-as-judge and RAGAS, and storing evaluation results for analytics.

---

## 2. Core Capabilities

### 2.1 Retrieval-Augmented Generation (RAG)
- Document ingestion from local directory.
- Embedding generation using SentenceTransformers.
- Vector storage in PostgreSQL with pgvector.
- Context retrieval using similarity search.
- Prompt governance using versioned prompts stored in PostgreSQL.
- LLM answer generation using GPT‑4o‑mini.

### 2.2 Automatic Evaluation (Every Query)
- LLM-as-judge scoring for correctness, groundedness, and completeness.
- RAGAS metrics for retrieval quality (faithfulness, relevance, precision, recall).
- Evaluation results stored in MongoDB Atlas.
- Evaluation returned to frontend with every answer.

### 2.3 Guardrails
- Input validation using Pydantic models.
- Output schema validation.
- Safety filters for sensitive or disallowed content.
- Hallucination detection using LLM-as-judge + RAGAS.

### 2.4 Multi-Agent Architecture
- Summarizer agent.
- Evaluator agent.
- RAG agent.
- Guardrail agent.
- Multimodal agent (audio transcription, image analysis).
- Agents orchestrated using LangGraph and Smolagents.

### 2.5 Memory Architecture
- PostgreSQL (pgvector) for long-term embeddings.
- ChromaDB for short-term session memory.
- MongoDB Atlas for:
  - Evaluation results
  - Agent logs
  - Multimodal metadata

### 2.6 LLMOps & Observability
- OpenTelemetry tracing.
- Structured logging.
- Metrics for:
  - Average judge score
  - RAGAS metrics over time
  - Prompt version performance
- Future: Phoenix or LangSmith integration.

3.3 Architecture Decisions (Updated Mar‑05‑2026)
1. Unified Router Structure
The backend now uses a single global router located at app/router.py.
	◦ Removed the previous `app/rg/router.py to avoid fragmented routing.
	◦ All RAG endpoints (/rag/query, /rag/ingest) are mounted through this unified router.
	◦ This ensures a single API entrypoint and simplifies future agent and workflow integration.
2. Typed API Response Model
Introduced the `RAGResponsePydantic model to enforce consistent API output.
	◦ Fields include: answer, sources, judge_score, ragas_scores, drift, `latencyms, `throughput
	◦ /rag/query now declares `responsemodel=RAGResponse for validation and OpenAPI documentation.
	◦ This aligns with the system’s requirement for transparent, evaluable responses.
3. Sync/Async Boundary Handling
The FastAPI endpoints remain synchronous (def), while the RAG pipeline is asynchronous (async def).
	◦ Endpoints call the async pipeline using `asyncio.run) to bridge the sync/async boundary.
	◦ This avoids event-loop conflicts and keeps the API surface simple for clients.
4. Corrected FastAPI Entrypoint
Updated main.py to import the unified router using:
from app.router import router
	◦ Ensures the correct router is loaded.
	◦ Prevents import errors after removing the old nested router.

5. Evaluation Storage Schema (MongoDB Atlas)
Collection: `evaluations`
{
  "question": "string",
  "answer": "string",
  "sources": ["string"],

  "judge_score": 0.0,
  "judge_justification": "string",

  "ragas_scores": {
    "faithfulness": 0.0,
    "answer_relevance": 0.0,
    "context_precision": 0.0,
    "context_recall": 0.0
  },

  "drift": {
    "embedding_shift": 0.0,
    "semantic_shift": 0.0
  },

  "latency_ms": 0.0,
  "throughput": 0.0,

  "created_at": "datetime"
}
Drift metrics track how the system’s behavior changes over time. Embedding drift shows when the meaning of stored vectors begins to shift, and semantic drift shows when the model’s answers start diverging from earlier patterns. These signals help identify when re‑embedding, prompt updates, or model adjustments are needed. Performance metrics such as latency and throughput capture how quickly the system responds and how well it handles load, which supports observability and capacity planning.

5.1. Evaluation Engine Integration (Updated Mar‑08‑2026)
   - LLM-as-judge fully integrated into RAGPipeline.
   - RAGAS evaluator wired to compute real metrics (faithfulness, relevance, precision, recall).
   - EvaluationRecord now stores real judge score, justification, RAGAS metrics, drift, and performance.
   - MongoEvaluationStore receives complete evaluation documents for analytics.
   - Observability layer added:
        • Structured logging (query start, completion, evaluation)
        • EvaluationMetrics for tracking judge score trends
        • OpenTelemetry tracing hooks for RAG queries
   - RAGPipeline now emits latency, throughput, and evaluation metadata for future dashboards.
